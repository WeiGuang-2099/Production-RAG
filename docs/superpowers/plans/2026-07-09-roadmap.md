## 3. Latency optimization (plan AFTER item 1 lands)

Both this and item 1 edit `app/core/pipeline.py:_retrieve_and_rerank`, so the
plan is written once item 1 is merged. Verified targets (2026-07-09 code):

- `VectorStore` is constructed per query; `QdrantVectorStore.from_existing_collection`
  opens a fresh Qdrant connection every time (`app/retrieval/vector_store.py:33-41`).
- `BM25Store.__init__` unpickles the whole index from disk on every query
  (`app/retrieval/bm25_store.py:63-75`) — cost grows with corpus size, so
  measure on the 30-paper corpus from item 2.
- Vector, BM25, and graph legs run serially (`app/core/pipeline.py:149-209`).

Approach: cached store factories (shared between ingest and query paths,
invalidated via the existing `clear_caches()`), parallelize the three
retrieval legs, then rerun `run_ablation.py --label` before/after on both
corpora and publish the latency table. Deliverable is a before/after story
with p50/p95 numbers, not just faster code.
